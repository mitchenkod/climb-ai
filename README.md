# Climb AI

Climb AI - это система для описания скалодромных стен, ручной разметки зацепов по фото и дальнейшей генерации маршрутов. Текущий MVP строится вокруг точной геометрии стены и ручной разметки: компьютерное зрение и YOLO остаются важным следующим направлением, но не являются основным рабочим сценарием на текущем этапе.

## Текущий фокус

Сейчас проект решает базовую продуктовую цепочку:

1. Создать зал (`Gym`).
2. Загрузить фото стены (`Wall`).
3. Задать геометрию плоскости стены (`Surface`): размеры, пиксельный размер изображения и угол наклона.
4. Разметить зацепы (`Hold`) вручную на изображении.
5. Указать для зацепа тип, качество по шкале 1-10 и направления работы.
6. Сгенерировать старт маршрута по сохраненным зацепам.
7. Выбирать маршрут из списка, добавлять следующие движения и анализировать вес позиций.

Архитектурная модель уже заложена с запасом:

```text
Gym -> Wall -> Surface -> Hold
              -> Route
```

Одна стена может содержать несколько плоскостей. Сейчас реализован простой сценарий "одна стена - одна плоскость", но модель рассчитана на дальнейшие конфигурации: переменный угол нависания, внутренние и внешние углы, несколько зон стены с разной геометрией.

## Что уже работает

- FastAPI backend с SQLite через SQLModel.
- Создание зала.
- Загрузка фото стены через статический frontend без `python-multipart`: изображение передается JSON/base64 и сохраняется в `frontend/static/images/`.
- Автоматическое создание `Wall` и первой `Surface` при загрузке фото.
- Геометрия плоскости: ширина, высота, размер изображения в пикселях, угол наклона.
- Нормализация координат зацепов: пиксели, нормализованные координаты и метры на плоскости.
- Ручная разметка зацепов кликом по изображению.
- Редактирование, перетаскивание и удаление зацепов.
- Тип зацепа, качество 1-10 и векторы направлений работы зацепа.
- Обновление параметров плоскости с пересчетом координат зацепов.
- Защита от изменения геометрии, если по стене уже создан хотя бы один маршрут.
- Создание стены через выбор четырехугольника рабочей области на фото и проекцию фото на реальную плоскость.
- Страницы `/static/walls`, `/static/wall/new`, `/static/wall/{id}`.
- Базовая генерация старта маршрута с учетом реалистичности позиции.
- Список маршрутов по стене с пагинацией.
- Выбор, отрисовка и удаление маршрутов.
- Итеративное добавление следующего движения (`next_move`).
- Эвристики оценки положения тела: качество зацепов, расстояния, треугольник, направление работы, компактность/растянутость с учетом угла нависания.
- Вероятностное добавление новой плохой ноги на поздних кандидатах движения с проверкой, что она не слишком улучшает предыдущие позиции.
- Модульные тесты для геометрии, загрузки стены, разметки зацепов, запрета изменения геометрии, скоринга позиций и маршрутов.

## Стек

- Backend: FastAPI, SQLModel, SQLAlchemy, Pillow.
- Frontend: статические HTML/CSS/JS файлы, canvas overlay.
- Database: SQLite по умолчанию.
- Vision backlog: OpenCV, YOLOv8.
- Infrastructure: Docker, Docker Compose.

## Рабочий сценарий ручной разметки

1. Открыть `http://localhost:8000/static/walls`.
2. Перейти в `New wall`.
3. Указать `Gym ID`. Если зала еще нет, создать его через API или Swagger.
4. Выбрать изображение стены.
5. Указать ширину, высоту и угол плоскости.
6. Отметить четыре угла рабочей области стены на фото.
7. Создать стену и перейти на страницу `/static/wall/{wall_id}`.
8. Кликать по фото для добавления зацепов.
9. Для каждого зацепа выбрать тип, качество и направления работы.
10. Перетаскивать выбранный зацеп для уточнения позиции.
11. Использовать `Update selected` для обновления параметров выбранного зацепа.
12. При необходимости изменить параметры стены через `Update wall params`.
13. После создания маршрута геометрию стены менять нельзя: backend вернет `409 Conflict`.

## Рабочий сценарий маршрутов

1. На странице стены сохранить размеченные зацепы.
2. Нажать `Generate route` для создания стартовой позиции.
3. Выбрать маршрут из списка `Routes`.
4. Использовать `Next move` в строке маршрута для итеративного добавления следующей ручной зацепки.
5. Смотреть рассчитанную позицию: текущий маршрут рисуется желтым/красным, новая оцененная позиция - зелеными окружностями.
6. При необходимости удалить маршрут через `Delete`.

Визуальные роли:

- `start` и `finish/top` - желтые окружности;
- `intermediate` - красные окружности;
- текущая рассчитанная позиция после `Next move` - зеленые окружности.

## API: ключевые операции

Создать зал:

```bash
curl -X POST http://localhost:8000/api/gyms \
  -H "Content-Type: application/json" \
  -d '{"name": "My Gym"}'
```

Загрузка стены выполняется frontend'ом через `POST /api/walls/upload` в JSON/base64 формате. Для ручной проверки проще использовать Swagger UI, потому что тело запроса содержит base64 изображения.

Основные endpoint'ы текущего MVP:

- `POST /api/walls/upload` - загрузить фото, создать стену и первую плоскость.
- `GET /api/walls/{wall_id}` - получить стену, плоскости и зацепы.
- `POST /api/walls/{wall_id}/holds` - добавить зацеп.
- `PATCH /api/holds/{hold_id}` - обновить параметры или координаты зацепа.
- `DELETE /api/walls/{wall_id}/holds/{hold_id}` - удалить зацеп.
- `PATCH /api/surfaces/{surface_id}/geometry` - обновить геометрию плоскости и пересчитать координаты зацепов.
- `POST /api/routes` - сгенерировать старт маршрута по `wall_id` или `surface_id`.
- `GET /api/walls/{wall_id}/routes` - получить маршруты стены с пагинацией.
- `DELETE /api/walls/{wall_id}/routes/{route_id}` - удалить маршрут.
- `POST /api/walls/{wall_id}/routes/{route_id}/next_move` - добавить следующее движение маршрута.

## Модель данных

- `Gym` - зал/скалодром.
- `Wall` - конкретная стена внутри зала.
- `Surface` - геометрическая плоскость стены.
- `Hold` - зацеп на плоскости.
- `Route` - маршрут по стене.
- `HoldInRoute` - связь маршрута и зацепов.
- `Movement` - вершина/движение графа достижимости.

`Hold` сейчас хранит:

- `x`, `y` - нормализованные координаты.
- `x_px`, `y_px` - координаты на изображении.
- `x_m`, `y_m` - координаты в метрах на плоскости.
- `hold_type` - тип зацепа.
- `quality` - удобство/качество по шкале 1-10.
- `force_vectors` - направления работы зацепа в JSON-строке.

## Оценка позиций и feature vector

Текущая генерация маршрутов строится вокруг объяснимого вектора признаков позиции. Для каждой позиции можно сохранять сырые признаки, веса и итоговую оценку.

Уже доступны признаки:

- типы зацепов и роли в маршруте: `start`, `intermediate`, `finish`, `foot_suggested`;
- качество зацепов 1-10;
- направления работы зацепов;
- координаты зацепов в пикселях, нормализованной системе и метрах на плоскости;
- угол наклона стены;
- количество контактов;
- средняя и максимальная удаленность от геометрического центра;
- расстояния между руками и ногами;
- штраф за слишком компактные или растянутые позиции, линейно зависящий от угла нависания;
- правило треугольника;
- соответствие направления работы зацепа направлению к центру позиции;
- признаки реалистичности: слишком высокая стартовая нога, слишком высокие руки, слишком дальний reach, использование `foot` рукой;
- агрегаты маршрута: количество зацепов, наличие топа, количество предложенных ног, score позиции/маршрута.

Следующий архитектурный шаг - вынести расчет признаков в стабильный слой `RouteFeatureExtractor` / `MovementEvaluation`, хранить версию схемы признаков и версию весов. Это позволит:

- пересчитывать сложность маршрутов без изменения исходных данных;
- подбирать коэффициенты вручную;
- обучать модель на маршрутах с известной сложностью;
- сравнивать разные версии скоринга.

## Известные ограничения

- Текущий frontend рассчитан на одну активную плоскость стены.
- SQLite и dev schema patching подходят для MVP, но не для production.
- CV/YOLO scripts пока не являются рабочим основным pipeline.
- Streamlit frontend не является актуальным основным интерфейсом.
- `backend/app.db` - локальный dev artifact и не должен использоваться как источник истины.

## Решение проблем

Если backend не запускается:

- проверьте, что зависимости установлены;
- проверьте, что порт `8000` свободен;
- выполните `python scripts/dev_bootsrap.py`.

Если интерфейс разметки не открывается:

- проверьте, что запущен backend;
- откройте `http://localhost:8000/static/walls`;
- проверьте консоль браузера и ответы API в Swagger UI.
